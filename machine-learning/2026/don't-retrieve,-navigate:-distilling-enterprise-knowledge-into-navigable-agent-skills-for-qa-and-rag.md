# Overview

**Corpus2Skill** is a two-phase framework that transforms a document corpus into a hierarchical **skill directory** that an LLM agent can navigate at inference time, eliminating the need for embedding-based retrieval entirely. Instead of treating retrieval as a black-box similarity search, the agent explicitly browses a filesystem-like hierarchy of summaries and documents to find relevant information.

> [!NOTE]
> "Make the hierarchy itself the primary interface for the LLM. Rather than searching *over* a hierarchy, the agent navigates *through* it."

**Applicable when**: An enterprise has a single-domain, curated knowledge base with atomic documents (0.5–3k characters each) and needs accurate, auditable QA over it (e.g., customer support, policy documentation).

**Not applicable when**: Documents are long and extractive (legal contracts), corpora are homogeneous tabular data, open-domain recall is required, or real-time corpus updates are needed.

# Background: Limitations of Retrieval-Based RAG

Traditional RAG pipelines embed all documents and perform k-nearest-neighbor lookup at query time. This approach has two structural weaknesses:

1. **Lost structure**: Embedding-based retrieval collapses the document corpus's organizational hierarchy into a flat pool of vectors, discarding topical groupings that help disambiguate queries.
2. **Passive consumption**: The LLM receives top-k chunks without any awareness of the broader corpus organization; it cannot backtrack or re-navigate if the retrieved passages are insufficient.

Hierarchical RAG systems such as **RAPTOR** address point 1 by building tree-structured summaries, but they still rely on embedding-based retrieval to select which nodes to surface—they retain the retrieval bottleneck.

**Corpus2Skill** removes retrieval entirely; the LLM agent navigates the hierarchy via tool calls.

# System Architecture

Corpus2Skill consists of two phases: **offline compilation** and **online serving**.

## Compile Phase

**Input**: A flat corpus of $N$ documents $\mathcal{D} = \{d_1, \dots, d_N\}$.  
**Output**: A hierarchical skill directory stored as a filesystem.

### Step 1 — Document Embedding

Each document $d_i$ is embedded into a dense vector $\mathbf{v}_i \in \mathbb{R}^{d}$ using a sentence encoder (Qwen3-Embedding in experiments).

### Step 2 — Iterative Hierarchical Clustering

K-Means clustering with branching ratio $p$ partitions documents into groups at each level. Clustering is applied recursively until each leaf cluster contains at most $p$ documents:

```
function build_hierarchy(docs, p):
    if |docs| <= p:
        return LeafNode(docs)
    clusters = KMeans(docs, k=p)
    children = [build_hierarchy(cluster, p) for cluster in clusters]
    return InternalNode(children)
```

For a corpus of $N = 6221$ documents with $p = 10$, this produces a tree of depth 3 requiring only $\approx 30$ summary files at the top two levels—demonstrating $O(\log_p N)$ navigation cost.

### Step 3 — LLM Summarization

Each internal node receives a **routing-oriented summary** generated by an LLM. Unlike generic summaries, these descriptions are written to help an agent decide whether to descend into a subtree: they enumerate typical question types, key entities, and topical coverage of the subtree.

**Input to LLM summarizer**: All child summaries (or document titles/snippets for leaf nodes).  
**Output**: A concise routing summary (a few sentences).

### Step 4 — Skill Tree Materialization

The resulting hierarchy is stored as a filesystem:

```
skills/
  commerce/              ← top-level skill (SKILL.md: routing summary)
    payments/            ← mid-level group (INDEX.md: child list + descriptions)
      account_mgmt/      ← leaf group (INDEX.md: document list)
        doc_0042.txt     ← raw document
```

Two file types are materialized:
- **SKILL.md**: Routing summary for a skill (top-level grouping)
- **INDEX.md**: List of child skills/documents with brief descriptions for navigation at any level

The total compilation cost for 6,221 documents: **6.5 minutes** on a 32-CPU server, **$5–$10** in LLM API calls, producing 665 navigation files and a 13 MB document store.

## Serve Phase

**Input**: A user query $q$.  
**Output**: An answer generated by an LLM agent after navigating the skill directory.

### Tool Interface

The agent has access to two tools:

| Tool | Description |
|------|-------------|
| `code_execution(path)` | Read a navigation file (SKILL.md or INDEX.md) to see routing summary or child listing |
| `get_document(doc_id)` | Retrieve the full text of a specific document |

### Agent Navigation Loop

```
1. Agent receives: query q + pre-loaded top-level skill names and descriptions
2. Agent selects a skill branch to explore
3. Agent reads INDEX.md files recursively (progressive disclosure)
4. Agent calls get_document(id) for relevant leaf documents
5. Agent synthesizes answer from retrieved documents
6. If evidence is insufficient, agent backtracks and explores alternative branches
```

> [!IMPORTANT]
> Navigation files (SKILL.md, INDEX.md) are small and loaded on-demand; full documents are only fetched when needed. This keeps per-step context overhead low.

### Navigation Patterns Observed

1. **Direct Descent**: Agent follows a single path from top-level skill → subgroup → leaf → document when the query maps cleanly to one subtree.
2. **Cross-Branch Exploration**: Agent visits multiple subgroups and synthesizes evidence from different branches when the query spans multiple topics.

Example: A query about switching business type navigates `commerce skill → payments ecosystem → account management documents`.

# Comparison with Related Methods

| Method | Hierarchy Built | Retrieval Mechanism | Agent Navigation |
|--------|----------------|---------------------|-----------------|
| BM25 | None | Sparse keyword | No |
| Dense Retrieval | None | Embedding k-NN | No |
| Hybrid (RRF) | None | BM25 + embedding | No |
| RAPTOR | Summary tree | Embedding k-NN over nodes | No |
| Agentic RAG | None | Multi-turn embedding | Yes (unstructured) |
| **Corpus2Skill** | **Skill directory** | **None** | **Yes (structured filesystem)** |

> [!TIP]
> NaviRAG, HCAG, and GraphRAG also build hierarchical structures but all retain embedding-based retrieval to select nodes. Corpus2Skill eliminates this step.

Key differences from RAPTOR specifically:
- RAPTOR uses **soft, embedding-based node selection** at query time; Corpus2Skill uses **explicit agent navigation**.
- RAPTOR's hierarchy is used as a retrieval index; Corpus2Skill's hierarchy is the primary interface.
- RAPTOR cannot backtrack across branches; Corpus2Skill agents can explore multiple branches.

# Experiments

- **Dataset (primary)**: WixQA — 6,221 enterprise customer-support articles, 200 test queries. Single-domain, atomic documents averaging ~1k characters.
- **Dataset (generalization)**: RAGBench — 10 subsets spanning diverse domains (Emanual, DelucionQA, HAGRID, CovidQA, FinQA, CUAD, TatQA, etc.)
- **Hardware**: 32-CPU server for compilation; API-based inference for serving
- **Models**: Claude claude-sonnet-4-6 as agent (serving), Claude Haiku 3.5 for ablation; Qwen3-Embedding for document embedding
- **Optimizer**: N/A (inference-only evaluation)

### WixQA Results

| Method | Token F1 | Factuality | Context Recall | Cost ($/query) |
|--------|----------|------------|----------------|----------------|
| BM25 | ~0.36 | ~0.62 | ~0.55 | ~$0.007 |
| Dense Retrieval | ~0.38 | ~0.65 | ~0.58 | ~$0.008 |
| Hybrid (RRF) | ~0.39 | ~0.67 | ~0.60 | ~$0.009 |
| RAPTOR | ~0.41 | ~0.70 | ~0.63 | ~$0.012 |
| Agentic RAG | ~0.38 | ~0.68 | ~0.61 | ~$0.015 |
| **Corpus2Skill** | **0.468** | **0.739** | **0.673** | **$0.089** |

Corpus2Skill achieves 23% higher Token F1 than the Agentic RAG baseline and the highest factuality and context recall, at the cost of being 7–13× more expensive per query.

### Generalization (RAGBench)

Corpus2Skill achieves the **highest macro-average F1 (0.350)** across all 10 RAGBench subsets, but with uneven performance:

- **Strong**: Atomic, topically coherent corpora (Emanual, DelucionQA, HAGRID)
- **Weak**: Long extractive documents (CUAD legal: F1=0.246, cost=$0.97/query) and homogeneous tabular data (TatQA: F1=0.246)

# Ablation Studies

| Variable | Range Tested | Effect on Token F1 |
|----------|-------------|-------------------|
| Branching ratio $p$ | 5, 10, 15, 20 | 0.457–0.469 (minimal) |
| Exploration budget (turns) | 5–20 turns | 0.458–0.468 (stable) |
| Serving model | Haiku vs Sonnet | Haiku: 91% of Sonnet F1 at 29% cost |

The system is robust to branching ratio choice and exploration budget. Using Claude Haiku instead of Sonnet as the navigation agent reduces cost significantly with modest quality loss.

# Limitations

1. **Cost**: $0.089/query vs $0.007–$0.012 for retrieval-only methods (7–13× higher).
2. **API constraints**: Limited to 8 top-level skills and 200 files per skill under the Anthropic Skills API.
3. **Hard clustering**: Documents assigned to a single cluster; multi-topic documents create routing bottlenecks.
4. **Incompatible corpus shapes**: Fails on long extractive documents (>7,500 chars), homogeneous tabular data, and open-domain pools where recall dominates precision.
5. **Batch compilation**: No incremental updates; adding new documents requires full recompilation.

# Practical Guidance

**Use Corpus2Skill when**:
- Corpus is a single domain with atomic documents (0.5–3k chars each)
- Answer quality and factual accuracy are more important than cost
- The corpus is relatively static (no frequent updates)
- Enterprise customer support, policy Q&A, curated knowledge bases

**Use standard RAG when**:
- Corpus contains long documents with extractive answers
- Corpus is heterogeneous or tabular
- Cost per query must be minimized
- Corpus changes frequently
